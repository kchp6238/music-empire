# 핵심 게임 루프 (Core Game Loop)

> GDD v0.1의 디자인 필라(음악이 먼저다, 능력치는 확률이다 등)를 전제로, 플레이어가 실제로 반복하게 될 루프를 시간 단위별로 정의한다.

## 1. 세션 루프 (한 판 앉아서 하는 흐름)

```
곡 작업(스튜디오/비트메이커) → 발매 → 반응 확인(커뮤니티/차트) → 스탯·자금 재투자 → 다음 곡
```

한 세션의 목표 단위는 **곡 1곡을 완성해서 발매까지 가는 것**이다. 프로토타입 기준 한 곡 작업에 필요한 조작(섹션 배치, 드럼/베이스/피아노 입력, 가사, 발매)은 숙련자 기준 5~15분, 초심자 기준 15~30분 정도로 설계한다. 세션은 강제 종료 시점이 없다 — "방치형 자동 반복 금지" 원칙에 따라 다음 곡 작업도 플레이어가 능동적으로 시작해야 한다.

## 2. 일 단위 (Day)

하루는 실시간 진행이 아니라 **플레이어 행동 기반 논리적 단위**다 (Football Manager식 "다음 경기"가 아니라 "다음 곡 작업"이 하루를 넘긴다는 개념). 하루에 플레이어가 할 수 있는 것:

- 곡 작업 계속 (섹션/트랙 편집)
- 발매 (1일 1회 제한 — 도배성 발매로 차트/팬 반응을 오염시키는 것을 방지)
- 커뮤니티 활동: 다른 아티스트(NPC + 실유저) 곡 감상, 좋아요/댓글/팔로우
- 협업 요청 발송/응답 (10번 시스템, MVP 이후)

발매 1일 1회 제한은 "능력치는 확률이다" 원칙과도 맞물린다 — 발매 빈도로 확률을 밀어붙이는 플레이를 막는다.

## 3. 주 단위 (Week) — 팬 반응·차트 갱신 주기

- **차트는 주간 집계**로 갱신한다. 매주 시작 시점에 지난 7일간 발매곡들의 `SongPerformanceScore` 누적치로 순위를 재계산.
- **오프라인 팬 이탈/유입**은 실시간으로 흐르지 않는다. 대신 플레이어가 접속할 때마다 "지난 접속 이후 경과 일수"를 기준으로 배치 계산한다:
  - `character_fan_loyalty.loyalty_score`가 임계값 이상인 페르소나는 발매가 없어도 서서히 신규 팬 유입에 기여(팬덤 자연 성장)
  - 장기간(예: 14일+) 미접속·미발매 시 `discoveryOpenness`가 낮은 페르소나부터 이탈 확률 증가
  - 이 계산은 **로그인 시 1회 배치**로 처리한다 (서버가 유저 없는 동안 크론으로 계산하지 않음 — MVP 단계에서 불필요한 서버 부담을 피함)
- **역주행(sleeper hit)**은 발매 시점 즉시 판정이 아니라, 발매 후 수일에 걸쳐 노출이 뒤늦게 붙는 이벤트로 확장 가능(현재 프로토타입은 발매 즉시 판정 — MVP 이후 시간차 역주행으로 고도화 예정, §12 참고).

## 4. 시즌 단위 (Season)

- 시즌은 **분기(3개월) 단위**로 정의한다. 시즌 종료 시:
  - 시즌 차트 명예의 전당 기록 (연도-분기별 Top 10 스냅샷)
  - `character_fan_loyalty` 소폭 정규화 (과도하게 누적된 충성도 완만화, 신규 유저 진입 장벽 완화)
  - 시즌 테마/이벤트 훅 지점 (MVP 이후 — 라이벌/트렌드 시스템이 확정되면 이 지점에 연동)

## 5. 오프라인 시간 처리 요약

| 항목 | 처리 방식 |
|---|---|
| 팬 이탈/유입 | 실시간 아님, 로그인 시 배치 계산 |
| 차트 순위 | 주 단위 배치 재계산 (매주 월요일 00:00 UTC+9 기준, 서버 크론 또는 다음 조회 시 lazy 계산) |
| 곡 반응(좋아요/댓글) | 실시간 (다른 유저가 실제로 감상할 때 즉시 반영) |
| 협업 세션 | 비동기 지원 — 상대가 오프라인이어도 초대/작업물 등록 가능, 실시간 동시 편집은 MVP 이후 |

## 6. 결정론적 성공 금지 원칙과 루프의 관계

이 루프에서 "재투자"는 스탯을 올려서 다음 곡의 **평균 기댓값**을 높이는 것이지, 성공을 보장하지 않는다. 따라서 플레이어가 체감하는 루프는 "그라인딩하면 무조건 이긴다"가 아니라 "그라인딩하면 확률이 유리해진다 + 여전히 재능/운/팬 매칭이 결과를 흔든다"가 되어야 한다. UI/UX(§3) 설계 시 이 확률적 피드백을 명확히 보여주는 것이 중요하다 (예: 발매 결과 화면에서 "reach 대비 반응률" 같은 분산 정보를 노출).
