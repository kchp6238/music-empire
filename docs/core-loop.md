# 핵심 게임 루프 (Core Game Loop)

> GDD v0.1의 디자인 필라(음악이 먼저다, 능력치는 확률이다 등)를 전제로, 플레이어가 실제로 반복하게 될 루프를 시간 단위별로 정의한다.

## 1. 세션 루프 (한 판 앉아서 하는 흐름)

```
곡 작업(스튜디오/비트메이커) → 발매 → 반응 확인(커뮤니티/차트) → 스탯·자금 재투자 → 다음 곡
```

한 세션의 목표 단위는 **곡 1곡을 완성해서 발매까지 가는 것**이다. 프로토타입 기준 한 곡 작업에 필요한 조작(섹션 배치, 드럼/베이스/피아노 입력, 가사, 발매)은 숙련자 기준 5~15분, 초심자 기준 15~30분 정도로 설계한다. 세션은 강제 종료 시점이 없다 — "방치형 자동 반복 금지" 원칙에 따라 다음 곡 작업도 플레이어가 능동적으로 시작해야 한다.

## 2. 일 단위 (Day) — 행동으로 흘러가는 게임 달력

**시간은 플레이어가 행동할 때만 흐른다.** 현실 시간과는 완전히 분리돼 있어서, 하루 종일 접속하지 않아도 게임 날짜는 그대로이고, 반대로 한 자리에서 몇 년치 커리어를 진행할 수도 있다. 방치해서 얻는 이득이 없으므로 GDD의 "방치형 루프 금지" 원칙과도 맞는다.

구현: `backend/app/services/time_service.py`의 `advance_days()`가 달력을 움직이는 **유일한 지점**이며, 시간에 연동된 모든 효과(팬 증감·수익·주간/시즌 결산·노화)가 여기에 매달려 있다.

| 행동 | 소모 일수 |
|---|---|
| 곡 발매 | 7일 (프레스·프로모·첫 주 차트) |
| 스탯 훈련 | 3일 |
| 한 주 쉬기 | 7일 |
| 연습생 트레이닝 | 5일 |
| 연습생 모집 | 1일 |
| 콘서트 | 2일 |

곡 편집(비트메이커 작업) 자체는 시간을 소모하지 않는다 — 창작에 시간 제한을 두면 오히려 "음악이 먼저다" 원칙을 해치기 때문이다.

> **초기 설계와 달라진 점**: 원안의 "발매 1일 1회 제한"은 삭제했다. 발매가 이미 게임 내 일주일을 소모하므로 별도 제한이 불필요해졌고, 현실 하루를 기준으로 하던 옛 구현은 게임 달력과 모순됐다.

## 3. 주 단위 (Week) — 팬 변동과 결산

게임 달력이 한 주 경계를 넘을 때마다 정산한다. 행동 한 번이 여러 주를 건너뛰면 건너뛴 주 수만큼 반복 정산되며, 같은 주가 두 번 정산되는 일은 없다 (`last_settled_week`로 추적).

- **팬 유입/이탈**: 충성도 높은 페르소나가 많을수록 자연 유입이 크고 이탈률이 낮아진다. 경과 **게임 일수** 기준.
- **스트리밍 수익**: 보유 팬 수 × 경과 일수에 비례한 패시브 수익.
- **명성 감소**: 활동이 없으면 주당 소폭 하락 — 가만히 있는 것에도 비용이 생긴다.
- **유행 변화**: 게임 주차 기준으로 장르/무드 트렌드가 로테이션된다 (`services/trends.py`).
- **연습생 성장**: 회사 소속 연습생은 시간이 지나면 스스로 조금씩 성장한다.

## 4. 시즌 단위 (Season)

**13주(약 분기)**가 한 시즌이다. 시즌이 끝날 때마다:

- **시즌 기록 스냅샷** 저장 (`season_records`): 나이·명성·자금·팬·누적 스트리밍·그 시점 최고 곡. GDD §14대로 **끝이 없는 게임**이므로 "달성 목표"가 아니라 **지나온 궤적의 기록**이다.
- **나이에 따른 능력 변화**: 32세를 넘기면 보컬 같은 수행 계열이 서서히 떨어지고(하한 30), 비즈니스·마케팅·작사 같은 판단 계열은 올라간다. 단 경험만으로는 **70까지만** 오르고, 그 위는 훈련해야 한다 — 시간만 보내도 스탯이 최대치가 되면 훈련이 무의미해지고 방치가 전략이 되기 때문이다.

## 5. 시간 처리 요약

| 항목 | 처리 방식 |
|---|---|
| 게임 날짜 | **행동할 때만** 진행 (현실 시간과 무관) |
| 팬 이탈/유입 · 스트리밍 수익 | 경과 **게임 일수** 기준, `advance_days()` 안에서 정산 |
| 유행(트렌드) | 게임 주차 기준 로테이션, 캐릭터마다 자기 시점의 트렌드를 본다 |
| 주간/시즌 결산 | 경계를 넘을 때 누락·중복 없이 정산 |
| 나이 | `birth_date`와 게임 날짜로 산출, 시즌마다 능력치에 반영 |
| 곡 반응(좋아요/댓글) | 실시간 (다른 유저가 실제로 감상할 때 즉시 반영) |
| 협업 세션 | 비동기 — 상대가 오프라인이어도 초대/수락 가능 |

> 접속하지 않은 동안에는 **아무 일도 일어나지 않는다.** 초기 설계의 "로그인 시 오프라인 배치 정산"은 제거했다 — 현실 시간으로 보상이 쌓이면 그 자체가 방치형 루프이기 때문이다.

## 6. 결정론적 성공 금지 원칙과 루프의 관계

이 루프에서 "재투자"는 스탯을 올려서 다음 곡의 **평균 기댓값**을 높이는 것이지, 성공을 보장하지 않는다. 따라서 플레이어가 체감하는 루프는 "그라인딩하면 무조건 이긴다"가 아니라 "그라인딩하면 확률이 유리해진다 + 여전히 재능/운/팬 매칭이 결과를 흔든다"가 되어야 한다. UI/UX(§3) 설계 시 이 확률적 피드백을 명확히 보여주는 것이 중요하다 (예: 발매 결과 화면에서 "reach 대비 반응률" 같은 분산 정보를 노출).
