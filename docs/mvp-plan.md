# MVP 개발 계획

## 1. MVP 목표 재정의

현재 프론트 프로토타입은 이미 "플레이 가능한 루프"를 증명했다 (캐릭터 생성 → 곡 작업 → 발매 → 다차원 평가 → 커뮤니티). 따라서 이번 MVP의 목표는 새 게임플레이가 아니라 **"영속성 + 최소 멀티유저"** 확보다.

**MVP 완료 기준(Definition of Done):**
1. 회원가입/로그인 후 캐릭터가 DB에 저장되고 새로고침해도 유지된다
2. 곡 작업물이 서버에 저장되고, 발매 시 점수가 **서버에서** 계산된다 (클라이언트 계산은 미리보기로만 사용)
3. 두 명 이상의 실제 유저가 서로의 발매곡을 커뮤니티 피드/차트에서 볼 수 있다
4. NPC 콘텐츠(5개 아티스트)는 시드 데이터로 유지되어 초기 콘텐츠 공백을 메운다

## 2. 주차별 마일스톤 (총 6주 가정)

| 주 | 목표 | 산출물 |
|---|---|---|
| 1주 | DB 스키마 확정 + 마이그레이션 | `db-schema.md` 기반 Alembic 초기 리비전, 로컬 Postgres 기동 |
| 2주 | 인증 + 캐릭터 API | `/auth/*`, `/characters` CRUD, Postman/OpenAPI로 수동 검증 |
| 3주 | 곡 저장 API + 서버 사이드 평가 엔진 | `/songs` draft CRUD, `scoring.py` 포팅, JS/Python 결과 diff 테스트 |
| 4주 | 프론트 로직 추출 + 로컬 동작 검증 | `frontend/src/lib`, `state`, `components` 분해, 백엔드 연동 없이 프로토타입과 동일 동작 확인 |
| 5주 | 프론트-백엔드 연동 | `lib/api/`, 캐릭터 생성/곡 발매가 실제로 DB에 반영, 새로고침 유지 확인 |
| 6주 | 커뮤니티 멀티유저 + 마무리 | 피드/차트가 실제 유저 데이터 반영, 팔로우 영속화, 회귀 테스트, 배포(Vercel+Railway) |

## 3. MVP 범위에서 명시적으로 제외

- 실시간 WebSocket 기능(콘서트 등) — REST만으로 충분한 범위까지만
- 협업 시스템(10번) — 초대/승인/기여도 분배 플로우 전체
- 회사 시스템(12번 GDD) — 연습생/영입/그룹/투어
- 라이벌 아티스트·트렌드 시스템, 마일스톤/업적 시스템 — **사용자 미확정 보류 항목** (GDD §15, 핸드오프 §5)
- 모바일 최적화 (ui-ux.md §4)

## 4. 리스크

| 리스크 | 대응 |
|---|---|
| JS↔Python 점수 로직 불일치로 밸런스 회귀 | Stage 4에서 고정 테스트 드래프트로 양쪽 출력 diff 검증 (implementation-order.md §5) |
| Postgres 로컬 환경 미구축(Docker 미설치 등) | `docker-compose.yml` 우선 제공, 없으면 로컬 Postgres 설치 가이드로 대체 |
| 단일 파일 → 모듈 분해 중 동작 누락 | Stage 2를 "백엔드 연동 전 로컬 동작 재현"으로 먼저 완료해 리팩터링 자체의 회귀를 조기 발견 |
