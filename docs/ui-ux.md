# UI/UX 설계

> 현재 프로토타입(`docs/reference_prototype.jsx`)의 화면 흐름을 기준으로 정식 정보구조(IA)를 문서화한다.

## 1. 화면 흐름 (Screen Flow)

```
인트로
  └─▶ 캐릭터 생성 (배경 9종 선택 + 아티스트명 입력)
        └─▶ 메인 3탭 구조 (TopBar로 상시 전환)
              ├─ 스튜디오 (곡 메타데이터: 제목/BPM/장르/무드/코드진행/제작모드/보컬소스)
              ├─ 비트메이커 (섹션 구성 + 드럼/베이스/피아노/기타 편집 + 가사 + 믹서 + 재생)
              └─ 커뮤니티 (피드 / 차트 탭, 팔로우)
```

TopBar는 항상 명성/자금/팬 수/발매곡 수를 상시 노출한다 (프로토타입 `TopBar` 컴포넌트 그대로 유지).

## 2. 정보 구조 (IA)

- **캐릭터 생성**: 배경 카드 그리드(9개) → 카드 선택 시 스탯/재능/명성/자금 프리셋 미리보기 → 아티스트명 입력 → 확정
- **스튜디오 탭**: 곡의 "메타데이터 계층" — 여기서 정한 값(BPM/장르/무드/코드진행/제작모드/보컬소스)이 비트메이커와 평가 엔진의 입력이 된다. 스튜디오는 비트메이커보다 상위 계층으로 취급 — 먼저 방문하도록 유도(제목 미입력 시 비트메이커 진입은 허용하되 발매 버튼은 비활성화).
- **비트메이커 탭**: 섹션 타임라인(인트로/벌스/코러스/브릿지/아웃트로 자유 배치) → 섹션 선택 시 해당 섹션 편집기(드럼 스텝시퀀서 + 베이스/피아노/기타 노트 레인 + 가사) → 하단 고정 믹서 패널 → 재생 컨트롤(섹션 미리듣기 / 전체 곡 재생) → 발매 버튼
- **커뮤니티 탭**: 피드(최신순, 내 곡 + NPC 5명 곡)와 차트(점수순 랭킹) 서브탭. 각 곡 카드에서 재생/좋아요/댓글 확인/팔로우 가능.

## 3. 발매 결과 화면 (신규 — 프로토타입엔 결과가 인라인으로만 표시됨)

발매 직후 별도의 "발매 리포트" 모달/화면을 신설할 것을 제안한다:
- 4대 속성 (완성도/독창성/대중성/실험성) 레이더 또는 바 형태
- 도달률(reach) 대비 반응 분포 — "몇 명에게 도달했고 그중 몇 %가 좋아했는가"를 시각화해 확률 기반 결과라는 점을 체감시킴
- 특수 이벤트 배지 (역주행 sleeper hit / 천재 이벤트 geniusEvent)
- 명성·자금·팬 변화량 (delta)

이 화면은 core-loop.md §6의 "확률적 피드백을 명확히 보여줄 것" 요구를 충족하기 위한 것으로, GDD의 "능력치는 확률이다" 원칙을 UI 차원에서 체감시키는 핵심 지점이다.

## 4. 모바일 대응

**MVP는 데스크톱 웹 우선으로 확정**한다. 근거:
- 비트메이커의 스텝 시퀀서/피아노 롤은 정밀한 클릭 입력이 필요해 터치 UX 재설계 비용이 큼
- 사용자 기술 스택 경험(React/Vite/Tailwind, Electron 미경험)상 모바일 앱 별도 트랙은 MVP 범위 밖
- TailwindCSS 반응형 유틸리티로 "최소 동작"(뷰만 가능, 편집은 데스크톱 권장 안내) 정도는 열어두되, 본격 모바일 최적화는 로드맵(§12) 후반부 항목으로 이연한다.

**설치형 웹앱(PWA)은 별도 네이티브 트랙 없이 확보했다.** `vite-plugin-pwa`로 매니페스트(`icons/`, `standalone` 표시 모드)와 서비스워커를 붙여, 데스크톱/모바일 브라우저 양쪽에서 "홈 화면에 추가" 설치가 가능하다. 캐싱 대상은 앱 셸(JS/CSS/폰트)뿐이며, `/auth`·`/characters`·`/songs`·`/community` 등 API는 별도 오리진(백엔드 포트)이라 서비스워커가 손대지 않는다 — 오프라인에서는 앱은 열리지만 게임 데이터는 로그인 후에만 갱신된다. 이는 Electron/React Native 같은 네이티브 앱 트랙을 대체하는 것이 아니라, 그 전 단계의 저비용 "설치 가능한 웹"을 제공하는 것이다.

| 공용 | `Fader`, `MiniBar`, `TopBar` | `components/shared/*` |
