# 확장 로드맵

> MVP(persistence + 최소 멀티유저) 완료 이후 진행 순서. GDD의 "미구현 시스템"(핸드오프 §4)을 기준으로 우선순위를 매겼다.

## 1. 협업 시스템 (GDD §10)
초대 → 승인 → 역할(작곡/작사/편곡/보컬/프로듀싱) 확정 → 비동기/동기 공동 작업 세션 → 발매 시 `contribution%` 기준 자동 수익 분배. `songs.collaborators` 필드(현재 프로토타입 개념 모델에 존재)를 정식 `song_collaborators` 테이블로 승격.

## 2. 회사 시스템 (GDD §12)
`companies`/`company_members`/`trainees` 테이블(db-schema.md §3에 자리만 예약됨) 활성화. 연습생 모집, 가수/프로듀서/작곡가 영입, 그룹 제작, 월드 투어, 해외 진출. "여러 아티스트의 곡 생산 파이프라인을 관리하는 메타 레이어"로 기존 5~10번 시스템을 재사용.

## 3. 경제 시스템 심화 (GDD §13)
현재는 발매 시 단순 delta만 계산. 수익원(스트리밍/공연/광고/팬클럽/앨범판매/라이선스)과 지출(급여/마케팅/장비/사옥운영)을 세분화한 원장(ledger) 테이블 도입. 회사 시스템 활성화 이후 손익계산 복잡도 증가하는 단계적 난이도 적용.

## 4. 팬 시간 경과 시뮬레이션 고도화
core-loop.md §3의 "로그인 시 배치 계산"을 넘어, 발매 후 수일에 걸친 지연 노출(역주행 지속성), 신곡 부스트 감쇠 곡선 등 시간 축 모델링 추가.

## 5. 차트/노출 알고리즘 고도화
현재(및 MVP) 차트는 주간 점수 합산 정렬. 순위 변동 히스토리, 신곡 부스트 가중치, 역주행 지속 판정을 별도 스코어링 레이어로 분리.

## 6. 온라인 연결 기능 확장 (GDD §11)
- 콘서트: 일정 예약형 소규모 동시 접속 이벤트 (WebSocket 채널 신설 — server-architecture.md §1에서 이미 자리 예약)
- 곡 거래(마켓플레이스)
- 아티스트 영입 요청/계약 플로우 (회사 시스템과 연동)

## 7. 보류 중 — 사용자 확정 필요 (임의 구현 금지)

아래 두 항목은 GDD/핸드오프 문서에서 "제안만 된 상태"로 명시되어 있다. **우선순위를 사용자에게 먼저 확인한 뒤** 착수한다.

- **라이벌 아티스트 & 유행 장르 변화(트렌드) 시스템**
- **마일스톤/업적 시스템**
