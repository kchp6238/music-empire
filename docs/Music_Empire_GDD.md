# Music Empire: The Music Life
## 게임 기획서 (Game Design Document) v0.1

> 문서 성격: 12개 산출물 중 **1번 — 완전한 게임 기획서**. 이후 핵심 루프 → UI/UX → DB → 서버 아키텍처 → MVP 계획 → 코드 순으로 이어지는 시리즈의 첫 문서다. 여기서 확정한 시스템 정의가 이후 모든 기술 문서의 기준이 된다.

---

## 0. 로그라인

플레이어가 음악 산업 안에서 **자신만의 인생**을 사는 오픈엔디드 라이프 시뮬레이션. 무명 뮤지션으로 시작해 전설이 되거나, 대형 기획사 대표로 제국을 세우거나, 인디 신에서 존경받는 아티스트로 남거나 — 정답은 없다. 게임의 중심은 언제나 **플레이어가 실제로 만든 음악**이다.

---

## 1. 디자인 필라 (Design Pillars)

| 필라 | 의미 | 구현 원칙 |
|---|---|---|
| 음악이 먼저다 | 회사·팬덤·차트는 결과이지 목적이 아니다 | 모든 시스템의 입력값은 "곡"이어야 한다 |
| 능력치는 확률이다 | 스탯은 성공을 보장하지 않는다 | 성공 판정은 항상 분포/가중치, 결정론적 임계값 금지 |
| 취향은 다원적이다 | 청취자마다 다른 곡을 좋아한다 | 청취자(팬) 단위로 취향 벡터를 갖는다 |
| 창작은 노동이다 | 곡은 자동 생성되지 않는다 | 플레이어의 실제 입력(코드/멜로디/선택)이 곡 데이터를 구성 |
| 성장에는 여러 길이 있다 | 뮤지션/프로듀서/경영자 루트가 공존 | 회사 시스템은 선택 사항, 강제 아님 |

### 절대 금지 사항 (Design Anti-Patterns)
- 스탯 임계값 기반 확정 성공 (예: 작곡력 90 이상이면 항상 히트)
- 결제 = 성장 (P2W)
- 방치형 자동 반복 클릭 루프
- 모든 팬이 같은 곡에 같은 반응을 보이는 단일 평가값

## 2. 레퍼런스와 차별점

| 참고작 | 가져오는 요소 | Music Empire의 차이 |
|---|---|---|
| Football Manager | 스탯·매치엔진·시즌 루프·이적 시스템 | "매치엔진" 대신 "청취 반응 엔진" |
| The Sims | 캐릭터 욕구/관계/커스터마이징 | 욕구 대신 "창작 동기·번아웃" 관리 |
| Stardew Valley | 시간 흐름 기반 성장, 느긋한 페이싱 | 계절 대신 "발매 사이클·차트 주기" |
| Cities Skylines | 시스템 간 상호 연결(교통↔경제↔인구) | 곡↔팬↔차트↔수익의 연쇄 반응 |

어떤 게임도 그대로 복제하지 않으며, "매니지먼트 시뮬레이션의 깊이 + 실제 창작 툴의 손맛"을 결합한다.

## 3. 시작 배경 (Starting Backgrounds)

강제 없이 선택 또는 커스텀 설정. 각 배경은 명확한 트레이드오프를 가진다.

| 배경 | 장점 | 단점 |
|---|---|---|
| 무명 음악가 | 완전한 자유, 낮은 리스크 | 인지도 0, 초기 수익 없음 |
| 유명 가수 | 초기 팬덤·수익 확보 | 대중의 기대치 부담, 실패 시 평판 타격 큼 |
| 신인 프로듀서 | 협업 네트워크 시작값 높음 | 본인 명의 곡 인지도 낮음 |
| 천재 작곡가 | 천재성 스탯 초기 보너스 | 사교성·경영 스탯 낮음, 상업성 이해 부족 |
| 연습생 | 트레이닝 시스템 접근, 성장 커리큘럼 | 계약 제약, 초기 자율성 낮음 |
| 인디 뮤지션 | 창작 자유도 최대, 팬 충성도 형성 유리 | 자금·유통망 취약 |
| 신생 기획사 대표 | 회사 시스템 조기 개방 | 자본 부담, 소속 아티스트 관리 난이도 |
| 대형 기획사 대표 | 풍부한 자본·인프라 | 관료적 제약, 주주/여론 압박 |
| 랜덤 인생 | 예측 불가한 조합(리플레이성) | 밸런스 예측 불가 |

각 배경은 **시작 스탯 프리셋 + 시작 자원(자금/인맥/계약 상태) + 고유 이벤트 풀**로 구현한다(세부 수치는 밸런싱 문서에서 확정).

## 4. 캐릭터 시스템: 스탯 vs 재능

### 4.1 실력 스탯 (훈련으로 성장, 상한 존재)
작곡 · 작사 · 편곡 · 보컬 · 연주(악기별) · 프로듀싱 · 믹싱/마스터링 · 비즈니스 감각 · 마케팅

### 4.2 재능 (Talent — 대부분 고정, 극히 느리게 변화)
천재성 · 창의성 · 음악적 감각 · 카리스마 · 노력(성장 속도 배율) · 리더십 · 운

**핵심 설계 원칙:** 실력 스탯은 "곡의 기술적 완성도"에, 재능은 "확률의 분포와 이벤트 발생"에 영향을 준다. 즉 스탯 100이라도 좋은 결과의 **평균**만 올리고, 재능(특히 운·천재성)이 **분산과 이례적 결과**(대박/대실패, 혁신적 장르 개척)를 결정한다.

### 4.3 천재성의 실제 역할
- 히트 확률 직접 상승 ❌
- 신규 장르·스타일 개척 이벤트 발생률 ↑
- 독창성(Originality) 스탯 상한 확장
- 특별 이벤트(영감의 순간 등) 트리거 확률 ↑

## 5. 음악 제작 시스템 (핵심 콘텐츠 — 최우선 개발)

### 5.1 작곡 워크플로우
BPM 설정 → 코드 진행 작성 → 멜로디 작성 → 드럼/베이스 작성 → 악기 선택 → 트랙별 분리 → (선택) MIDI 임포트/익스포트 → 피아노 롤 편집

곡은 최종적으로 다음 속성을 가진 데이터로 귀결된다:
- 장르, 무드(태그 다중 선택)
- 완성도(Craft) — 기술적 마감도, 실력 스탯 기반
- 독창성(Originality) — 재능 기반, 장르 관습에서의 이탈도
- 대중성(Accessibility) — 코드/멜로디 패턴의 익숙함 정도
- 실험성(Experimental) — 독창성과 대중성의 트레이드오프 축

### 5.2 초보자 지원 모드
자동 코드 추천, 분위기 기반 프리셋, 장르 추천, 멜로디 제안 — "음악을 몰라도 완주 가능"을 보장하는 안전망. 단, 지원 모드로 만든 곡은 독창성 상한이 낮게 설정되어 상단 밸런스를 해치지 않는다.

### 5.3 전문가 모드
멀티트랙 편집, 믹서, EQ, 리버브, 컴프레서, 자동 화성 분석 — 실력 스탯이 높은 플레이어에게 더 정밀한 제어권을 부여. 전문가 모드 사용 자체가 완성도 상한을 높인다.

### 5.4 곡 데이터 모델(개념)
```
Song {
  id, ownerId, title, genreTags[], moodTags[],
  bpm, key, structure(verse/chorus 등),
  craft: 0-100,        // 완성도
  originality: 0-100,  // 독창성
  accessibility: 0-100,// 대중성
  experimental: 0-100, // 실험성
  productionMode: "beginner" | "expert",
  collaborators: [{userId, role, contribution%}],
  vocalSource: "self" | "ai" | "collab" | "npc",
  releaseState, releaseDate,
  ...(세부 필드는 4번 산출물 DB 설계에서 확정)
}
```

## 6. 보컬 시스템
| 방식 | 특징 |
|---|---|
| 직접 녹음 | 플레이어 보컬 스탯 반영, 가장 낮은 비용, 상한은 스탯에 종속 |
| AI 보컬 | 즉시 이용 가능, 비용 발생, "감성/개성" 요소 페널티(팬 반응에서 일부 팬층 감점) |
| 다른 플레이어와 협업 | 실제 협업 시스템 연동, 기여도·수익 분배 발생 |
| NPC 보컬 고용 | 계약금/러닝개런티, NPC 유명도에 따라 곡 노출도 보너스 |

## 7. 공개·유통 시스템
곡 업로드 → 앨범 구성 → 플레이리스트 등록. 소셜 기능: 좋아요, 댓글, 팔로우, 저장, 공유.

---

## 8. 평가 시스템 (다차원 반응 엔진)

좋아요 총합 같은 단일 지표를 쓰지 않는다. 아래 요소의 **가중 조합**으로 곡의 성과를 산출한다(정확한 가중치는 밸런싱 대상, 아래는 설계 개념):

```
SongPerformanceScore =
    w1 * ListenCount +
    w2 * CompletionRate +      // 완청률 — 곡 품질의 핵심 신호
    w3 * RepeatPlayRate +      // 반복재생 — 진짜 좋아함의 신호
    w4 * SaveRate +
    w5 * ShareRate +
    w6 * CommentSentiment +
    w7 * FanAffinityMatch      // 팬 취향-곡 속성 매칭도
```

같은 곡도 **어떤 팬 풀에게 도달했는가**에 따라 이 값이 크게 달라진다 — 이는 8번(평가)과 9번(팬) 시스템이 상호작용하는 지점이다. 노출 알고리즘(차트/추천) 설계 시 "고품질 저노출 곡의 역주행" 이벤트를 위한 여지를 남긴다.

---

## 9. 팬 시스템

각 팬(NPC 청취자)은 **취향 벡터**를 가진다: 선호 장르/무드에 +가중치, 비선호에 -가중치.

```
FanProfile {
  genrePreference: { ballad: 0.8, edm: -0.6, hiphop: 0.3, ... },
  moodPreference: { emotional: 0.7, aggressive: -0.4, ... },
  loyaltyToArtist: {...},   // 아티스트 고정 팬덤 형성 시 사용
  discoveryOpenness: 0-1    // 새로운 장르 시도 개방성 — 낮으면 편식형 팬
}
```

곡의 (genreTags, moodTags, originality 등)과 팬의 취향 벡터 간 **코사인 유사도류 계산**으로 개별 반응을 산출 → 8번의 `FanAffinityMatch`에 반영. 이 구조 덕분에 "잘 만든 곡이 대중적으로 실패"하거나 "단순한 곡이 대박"나는 결과가 자연스럽게 나온다(타겟 팬 풀과의 매칭 여부가 실력보다 결과를 크게 좌우할 수 있음).

팬은 성장(신규 유입)·이탈(흥미 상실)하며, 아티스트에 대한 `loyaltyToArtist`가 축적되면 취향이 다소 안 맞는 곡도 관용적으로 받아들이는 버퍼 역할을 한다.

---

## 10. 협업 시스템
역할: 작곡 / 작사 / 편곡 / 보컬 / 프로듀싱. 각 참여자의 `contribution%`을 기록하고, 발매 후 수익을 해당 비율로 자동 분배한다. 협업 성사는 초대 → 승인 → 역할 확정 → 공동 작업 세션(비동기/동기 모두 지원) 흐름으로 처리한다.

---

## 11. 온라인 구조: "개인 세계 + 온라인 연결"
전원이 하나의 실시간 세계를 공유하는 MMORPG가 아니다. 플레이어는 **자신의 인스턴스**에서 시간을 진행시키고, 다음 지점에서만 다른 플레이어와 연결된다:
- 감상/댓글/좋아요 (비동기 소셜)
- 협업 (초대 기반)
- 차트 경쟁 (집계 기반, 실시간 대결 아님)
- 콘서트 (일정 예약형 이벤트, 소규모 동시 접속)
- 곡 거래 (마켓플레이스형)
- 아티스트 영입 (회사 시스템과 연동, 요청/계약 흐름)

이 구조는 서버 부담을 크게 낮추면서도 "혼자 하는 게임처럼 느껴지지 않게" 하는 절충안이다.

---

## 12. 회사 시스템 (후반 콘텐츠)
게임 시작부터 필수는 아니며, 플레이어가 성장 후 자율적으로 설립. 기능: 연습생 모집, 가수/프로듀서/작곡가 영입, 그룹 제작, 월드 투어, 해외 진출. 회사는 "여러 아티스트의 곡 생산 파이프라인을 관리하는 메타 레이어"로, 5~10번 시스템을 재사용하되 소유/계약 관계가 추가된다.

## 13. 경제 시스템
| 수익원 | 지출 |
|---|---|
| 스트리밍 | 직원 급여 |
| 공연 | 마케팅 |
| 광고 | 장비 |
| 팬클럽 | 연습생 육성 |
| 앨범 판매 | 사옥 운영 |
| 라이선스 판매 | (회사 시스템 활성화 후) |

초반(솔로 아티스트 단계)은 단순 수익/지출 구조로, 회사 시스템 활성화 후 손익계산 복잡도가 증가하는 **단계적 경제 난이도**를 적용한다.

---

## 14. 엔드게임
고정 엔딩 없음. "성공"의 정의 자체가 플레이어마다 다르다: 세계적인 가수, 전설적인 작곡가, 최고의 프로듀서, 대형 엔터 대표, 인디 음악의 왕, 차트 지배자, 음악 산업의 혁신가 등. 시스템적으로는 각 루트에 대한 **레퍼런스 지표(명성/자산/비평/혁신도)**를 추적해 플레이어가 자신의 진행 상황을 여러 축으로 확인할 수 있게 한다.
